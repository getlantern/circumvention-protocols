# VLESS + REALITY

## TL;DR

VLESS payload over a REALITY transport. The REALITY server is a TCP
relay to a real cover site (e.g. `dl.google.com`) that intercepts the
TLS handshake just long enough to recognize an AEAD-sealed marker in
the client's SessionID; if it's there, the server completes a TLS
handshake with the client using a freshly-signed ephemeral cert
(under an auth key derived via ECDH from the client's TLS key share),
and only then is the connection a proxy. Otherwise the server is just
a TCP relay to the real cover site and the probe completes a real
handshake with the real cover site's real cert.

## Threat Model

REALITY targets the same TLS-based detection vectors as samizdat /
Reflex, but with a distinct evasion mechanism:

- **SNI extraction**: SNI is the real cover domain. Block it and you
  block whatever you cover behind (`google.com`, `microsoft.com`,
  etc.). Picking high-collateral cover sites is the strategy.
- **JA3 / JA4**: ClientHello is generated by [uTLS](https://github.com/refraction-networking/utls)
  with a configurable browser fingerprint (Chrome by default).
- **Cert reputation**: the cert returned to a probe is the **real**
  cert from the real cover site (REALITY relays the handshake from
  the real server). To an authenticated client, the cert is an
  ephemeral cert signed inside the same TLS session — never seen on
  any other connection.
- **Active probing**: a probe that connects to the proxy IP and
  speaks a normal ClientHello gets a normal TLS session with the real
  cover site (cert, ServerHello quirks, HTTP responses, all real).
- **TLS-over-TLS detection**: VLESS is a thin framing protocol over
  the encrypted TLS stream — there's no nested TLS handshake.

It does **not** defend against IP-reputation blocking; the proxy IP
still has to be reachable by the censored client.

## Wire Format

```
Censored client                            REALITY relay (= proxy server)              Real cover site
  |                                              |                                          |
  |  TCP SYN                                     |                                          |
  |  TLS 1.3 ClientHello (uTLS Chrome) ------->  |                                          |
  |    SNI = chosen cover (e.g. google.com)      |                                          |
  |    SessionID = AEAD-sealed marker            |                                          |
  |    KeyShare = ephemeral X25519 pubkey        |  Server starts relaying packets to       |
  |                                              |  real cover. While doing so, it          |
  |                                              |  inspects the ClientHello to validate    |
  |                                              |  the SessionID marker.                   |
  |                                              |                                          |
  |                                              |  -- forward ClientHello ---------------> |
  |                                              |                                          |
  |                                              |    Auth path (marker valid):             |
  |                                              |       drop the connection to real cover. |
  |                                              |       Generate temp cert signed with the |
  |                                              |       derived auth key. Run TLS server   |
  |                                              |       toward client with that cert.      |
  |                                              |                                          |
  |                                              |    No-auth path (marker invalid):        |
  |                                              |       just keep relaying TCP both        |
  |                                              |       directions. Client gets the real   |
  |                                              |       site's real cert.                  |
  |                                              |                                          |
  |  <-- ServerHello + cert + Finished --------- |                                          |
  |     auth: temp cert signed by AuthKey        |                                          |
  |     no-auth: real cover-site cert            |                                          |
  |                                              |                                          |
  |  Client checks the cert it received:         |                                          |
  |    temp + AuthKey-verified -> proceed as     |                                          |
  |        a proxy. VLESS frames over the        |                                          |
  |        encrypted TLS stream.                 |                                          |
  |    real cover-site cert -> enter "spider"    |                                          |
  |        mode. Issue HTTP GETs to the cover    |                                          |
  |        domain over the same conn so the      |                                          |
  |        client looks like a real browser.     |                                          |
  |    invalid cert -> TLS alert, close.         |                                          |
```

## Cover Protocol

To a passive observer: HTTPS to the cover site, with the cover site's
real cert. Identical SNI, identical cert chain, identical TLS quirks
to a real visit. To a censor that watches the **whole** connection,
the steady-state encrypted traffic still has the structural shape of
"one long-lived HTTPS connection," which matches HTTP/2-using browser
sessions; REALITY doesn't try to fake browsing patterns at higher
layers.

## Authentication

Embedded in the TLS 1.3 ClientHello via the SessionID field.

### SessionID layout (32 bytes)

From [`Xray-core/transport/internet/reality/reality.go:140-148`](https://github.com/XTLS/Xray-core/blob/1dbafe6/transport/internet/reality/reality.go#L140-L148):

```
[0]      Xray Version_x
[1]      Xray Version_y
[2]      Xray Version_z
[3]      0  (reserved)
[4:8]    current Unix timestamp (uint32 big-endian)
[8:16]   shortId (pre-shared, configurable per client)
[16:32]  filled in by AEAD seal (see below)
```

### AuthKey derivation

1. Client generates an ephemeral X25519 keypair as part of the
   standard TLS 1.3 key-share extension. (REALITY also supports the
   `X25519MLKEM768` post-quantum group.)
2. Server's static X25519 **public key** is pre-shared with the
   client (`config.PublicKey`).
3. Client computes the ECDH shared secret using its **ephemeral
   private** and the **server's static public**:
   `AuthKey = ECDH(ephemeralPriv, serverStaticPub)`.
4. `AuthKey = HKDF-SHA256(IKM=AuthKey, salt=hello.Random[:20], info="REALITY")` (the `info` constant is the literal ASCII string `REALITY` — see `reality.go:167`).

### AEAD seal

```
aead = AES-GCM(key=AuthKey)
ciphertext, tag = aead.Seal(
    plaintext = hello.SessionId[:16],
    nonce     = hello.Random[20:32],   // 12 bytes
    aad       = hello.Raw,             // the entire ClientHello bytes
)
hello.SessionId = ciphertext || tag    // 16 + 16 = 32 bytes
```

The whole ClientHello is the AAD, so any modification by a censor
voids the tag. The nonce is part of the random bytes the client is
already sending in `ClientHello.random` — no extra fields, no
distinguishable bytes outside what a normal Chrome ClientHello has.

### Server-side verification

The REALITY server (in the [`xtls/reality`](https://github.com/XTLS/REALITY) package, a fork of `crypto/tls`) re-derives the same AuthKey from its own private key and the client's
key share, decrypts the SessionID, validates the timestamp +
shortId + version stamp, and verifies the AEAD tag against the
ClientHello bytes. If valid, the server intercepts the rest of the
handshake — it generates an **ephemeral cert** signed with the
AuthKey-derived signing key and serves that to the client. If
invalid, the server keeps relaying bytes to the real cover site and
the client sees the real cert.

### Client-side outcomes

`reality.go:183-209` distinguishes three results for the cert the
client received during the handshake:

1. **Ephemeral cert that verifies under AuthKey** → connection is
   trusted; proceed with VLESS framing over the encrypted stream.
2. **Real cover-site cert** → connection failed REALITY auth (or got
   intercepted upstream). Enter **spider mode**: spawn an HTTP/2
   client over the same connection and fetch URLs at `spiderX` paths
   on the cover domain to behave like a real browser. Don't proxy.
3. **Invalid cert** → close with a TLS alert.

## Probe Resistance

Two-layer:

1. **Default behavior is byte-level relay.** A passive probe (any TCP
   connection that doesn't carry a valid REALITY marker) is just a
   conduit between the probe and the real cover site. The probe
   completes a real TLS handshake with the real cover site's real
   cert; it talks HTTP to the real cover site. There is no
   probe-detection-and-fallback step, because **proxy-mode is itself
   the special case** — relay-mode is the default.
2. **Cert-distinction protects against MITM redirection.** A censor
   that intercepts a real client's connection and forwards it to the
   real cover site (instead of to the proxy) will deliver the real
   cert. The client's spider-mode response means even an *intercepted*
   real-client connection produces traffic indistinguishable from a
   normal browser hitting the cover domain.

This is materially different from samizdat / Trojan / Reflex, all of
which run the proxy logic by default and "fall back" to real-server
masquerading on failure. REALITY runs the relay by default and "falls
forward" into the proxy logic on success.

Key files:

- [`Xray-core/transport/internet/reality/reality.go`](https://github.com/XTLS/Xray-core/blob/1dbafe6/transport/internet/reality/reality.go) — client side. Notable: SessionID build at lines 140–148; ECDH + HKDF + AEAD seal at lines 152–175; spider-mode fetcher at lines 183–250.
- [`Xray-core/proxy/vless/`](https://github.com/XTLS/Xray-core/tree/1dbafe6/proxy/vless) — the VLESS payload protocol that rides inside REALITY.
- [`xtls/reality`](https://github.com/XTLS/REALITY) — the server. Fork of Go's `crypto/tls` with hooks to validate the SessionID, generate an ephemeral cert, and sign it with the derived auth key. Fallback to relaying happens inside the modified TLS state machine.
- [VLESS-XTLS-uTLS-REALITY example config](https://github.com/XTLS/REALITY/blob/9234c77/README.md) — server `target` (cover site `host:port`), `serverNames` (allowed SNI list), `privateKey` (server's X25519 static), `shortIds`. Client: `password` (server's X25519 public, base64), `shortId`, `serverName`, `fingerprint = "chrome"`, `spiderX` (initial spider path).

Notable post-2023 additions visible at `1dbafe6`:

- Optional ML-DSA-65 (post-quantum signature, `mldsa65Seed` /
  `mldsa65Verify`) for an additional auth signature over the cert
  beyond the X25519 AuthKey path.
- Optional X25519MLKEM768 hybrid key exchange (post-quantum) for the
  underlying TLS key share — the AuthKey still derives from the
  ECDH side of the hybrid.
- `limitFallbackUpload` / `limitFallbackDownload`: optional per-conn
  rate limits on the fallback (relay) path. README explicitly warns
  these are **a fingerprint** if not randomized — operators must
  jitter the parameters.

## Known Weaknesses

- **TLS record fragmentation** (e.g. [Niere et al. 2023, ACM CCS poster — `2023-niere-poster` in the public corpus](../circumvention-corpus/corpus/papers/2023-niere-poster.yaml)) operates at a layer above REALITY's mechanism. The poster discusses TLS-using transports including REALITY as candidates that benefit from layering record-level fragmentation on top.
- **Cover-site selection is operationally load-bearing.** The README's recommended cover-site criteria (foreign site, TLS 1.3 + H/2, encrypts ServerHello-onwards, OCSP stapling, IP-similarity, no host-based redirects) directly determine the cover quality. A blocked or unstable cover site collapses the cover.
- **Fallback rate-limit fingerprint**: `limitFallbackUpload` /
  `Download` are explicitly called out in the upstream README as a
  fingerprintable feature unless every operator independently
  randomizes them.
- **Behavioral signature of "one long-lived HTTPS conn carrying a lot of bytes"** — REALITY doesn't try to mimic browser tab churn or page-load bursts. Higher-layer behavioral DPI is out of scope.
- **The static server X25519 key is the long-term secret**; if it
  leaks, every recorded handshake retroactively reveals the proxy.
  Standard "rotate keys, revoke shortIds" hygiene applies.

## Deployment Notes

- Operationally common in the Iran / Russia / China circumvention
  community; the most-deployed REALITY-using stack outside Lantern.
- Picks heavily on cover-site choice — the README has a long
  guidance section on how to pick `target` (TLS 1.3 + H2 support,
  IP locality to the proxy, encrypted ServerHello-onwards, no
  host-based redirect, OCSP stapling, etc.).
- Interacts with `xtls-rprx-vision` (a separate XTLS flow that
  intercepts inner-TLS records to splice them rather than re-encrypt
  — a different evasion concern; not what makes REALITY work).
- Per the upstream README, REALITY can be paired with proxy
  protocols other than VLESS, but the team **does not recommend it**
  because non-VLESS payloads have observable TLS-in-TLS structural
  features that aren't present in raw VLESS framing.

- Related protocols (this catalog):

    *terminates* TLS itself with the proxy's own real cert — REALITY
    *intercepts* TLS toward a real cover site.
  - `reflex` — Lantern's role-reversal alternative. Different evasion
    primitive entirely (no ClientHello in the censored direction at
    all), but covers an overlapping threat surface (SNI / JA3-JA4 /
    fully-encrypted-traffic detection).
  - (TBD in this catalog) `trojan` — TLS-to-real-server pattern; the
    proxy operator owns the real cert and serves a fallback web app
