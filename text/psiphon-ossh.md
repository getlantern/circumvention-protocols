# Psiphon OSSH (Obfuscated SSH)

## TL;DR

The Psiphon tunnel family's foundational layer. An RC4-stream
obfuscation around a normal SSH session, keyed via HKDF from a
pre-shared "keyword" + a 16-byte random seed sent in the clear in
the first 24 bytes ("preamble"). The magic value `0x0BF5CA7E` lets
the server probabilistically detect a real OSSH client. After
handshake, the connection is just SSH inside RC4. Recent versions
add an **OSSH prefix** transform layer that prepends arbitrary
configurable bytes (HTTP-like, DNS-like, etc.) before the preamble
to defeat first-packet entropy detection.

The upstream source comment is unusually direct: *"New protocols
and schemes should not use this obfuscator."* OSSH stays in
deployment because every other Psiphon variant inherits it; the
modern variants (TLS-OSSH, MEEK-OSSH, etc.) get their evasion
properties from the **wrapping** layer, not from OSSH-alone.

## Threat Model

OSSH-alone, without a prefix or wrapping, is the canonical "look-
like-nothing" Psiphon protocol — it shares the obfs4-class threat
profile and the same fundamental weakness:

- **Defeats simple SSH-banner DPI** — the server doesn't send the
  `SSH-2.0-...` banner in the clear; the obfuscator runs first.
- **Defeats keyword-based protocol matchers** without the shared
  keyword.
- **Does NOT defeat fully-encrypted-traffic detection** (USENIX
  Security 2023). The wire is high-entropy from the first byte
  (16-byte random seed → 4-byte mostly-zero-but-encrypted magic →
  4-byte length → encrypted padding → encrypted SSH). This is
  exactly the FET classifier's target. Bare OSSH is blocked
  broadly by GFW + several other censors.
- **The OSSH prefix layer is the response.** When configured, the
  prefix prepends a transform-defined byte sequence (e.g. an
  HTTP-request-shaped prefix, a DNS-query-shaped prefix) so the
  first packet matches a known protocol's signature. This moves
  OSSH out of the FET classifier's "no protocol matched" bucket.
  The prefix spec is server-side tactics-controlled; clients pull
  the spec and apply the matching transform.

## Wire Format

### Without prefix (legacy / uncensored networks)

```
+----------------+--------------------+---------------+----------------+--------------------+
| 16 B           | 4 B                | 4 B           | up to 8192 B   | rest of conn       |
| seed           | magic 0x0BF5CA7E   | padding len   | random padding | obfuscated SSH     |
| (cleartext)    | (encrypted under   | (encrypted)   | (encrypted)    | (RC4-encrypted)    |
|                |  derived RC4 key)  |               |                |                    |
+----------------+--------------------+---------------+----------------+--------------------+
\__ 16-byte _/  \____________ 8-byte preamble header ____________/   \__ session bytes __/
   seed             ("PREAMBLE_HEADER_LENGTH = 24")
```

The seed is sent in the clear; both sides compute the RC4 key and
nonce as:

```
key  = HKDF-SHA1(seed || keyword) [iterated 6000 times — OBFUSCATE_HASH_ITERATIONS]
ivCS = HKDF derivative for "client_to_server"
ivSC = HKDF derivative for "server_to_client"
```

(See `obfuscator.go:46-49`. Hash count and labels are fixed
constants for compatibility.)

The server probabilistically rejects probes by validating the
magic value `OBFUSCATE_MAGIC_VALUE = 0x0BF5CA7E` after RC4-
decrypting the four bytes following the seed. A wrong keyword
produces a wrong magic with probability 1 in 2^32, so probes are
filtered out — but as the upstream comment warns, *"the magic value
provides only weak authentication due to its small size."*

After the preamble, the byte stream is just SSH. Standard SSH key
exchange (typically Curve25519-SHA256) then takes over and
establishes real session keys; OSSH's RC4 is only the
obfuscation-of-the-handshake layer.

### With OSSH prefix (modern / FET-resistant)

```
+--------------------------------+----------------+--------- ...
| transform-defined prefix bytes | preamble       |  (rest)
| (e.g. HTTP request, DNS query) | as above       |
+--------------------------------+----------------+--------- ...
```

The prefix is generated by applying a `transforms.Spec` (a
parameterized byte-replacement / regex-based generator) to a
deterministic seed. Server-side, the listener buffers up to
`PREFIX_MAX_LENGTH = 65536` bytes, scans for the prefix
**terminator** (a 16-byte sentinel), and strips the prefix before
handing the rest to the OSSH preamble parser.

A client picks the prefix spec from a per-region list pushed by
Psiphon tactics; an `OSSHPrefixHeader` field embedded in the
preamble tells the server which spec the client used (so the server
can re-derive the matching transform parameters for response-side
shaping).

## Cover Protocol

OSSH-alone has no cover. With a prefix, the **first packet** looks
like whatever protocol the prefix spec generates — but only the
first packet; steady-state is still the high-entropy
RC4-over-SSH stream, so a censor that fingerprints by sustained
flow shape can still recover signal.

For a real cover protocol on the entire stream, OSSH is wrapped
inside a transport — e.g. **TLS-OSSH** runs OSSH inside a TLS 1.3
connection (so steady-state is real TLS records); **MEEK-OSSH**
runs OSSH inside HTTP/HTTPS message bodies (steady-state is real
HTTP); **QUIC-OSSH** runs OSSH inside QUIC streams. The wrapping
protocol is what most contemporary deployments rely on; OSSH-alone
is mostly a legacy-compatibility option.

## Authentication

Two layers, both inheriting from SSH conventions.

- **Pre-shared "keyword"**: configured per-server, distributed via
  the Psiphon tactics system. Used to derive the RC4 key. Knowledge
  of the keyword is what makes a connection's magic value verify.
- **SSH server key**: standard SSH public-key authentication —
  the server presents a public key during the SSH key exchange,
  and the client validates it against a pinned fingerprint stored
  in its server entry. The keyword authenticates the obfuscator;
  the SSH key authenticates the actual server identity end-to-end.

Client authentication: standard SSH password or key authentication,
with credentials provisioned by Psiphon (often a sponsor-specific
SSH password embedded in the client config).

## Probe Resistance

- Magic-value check filters most stray TCP probes — without the
  shared keyword, the probe's RC4-decrypted magic doesn't match
  and the server drops the connection (with a randomized delay,
  matching the obfs4-style "make probing expensive" pattern).
- Replay defense: the obfuscator's seed is logged in a
  `SeedHistory` cache; replayed seeds are rejected. (See
  `seedHistory.go`; default policy enforced strictly when
  `DisableStrictHistoryMode = false`.)
- Once a connection passes the magic check, the SSH handshake's
  own server-pubkey authentication catches MITM attempts.

The probe-resistance story is the same as obfs4's — strong
against active probes that don't know the secret, weak against
passive FET-style classifiers. The OSSH prefix layer is the
explicit acknowledgement of that gap.

## Known Weaknesses

The upstream source comment in `obfuscator.go:80-89` is the
definitive statement, quoted in full:

> *Limitations:*
> - *The RC4 cipher is vulnerable to ciphertext malleability and
>   the "magic" value provides only weak authentication due to its
>   small size. Increasing the size of the magic field will break
>   compatibility with legacy clients.*
> - *The RC4 cipher does not provide integrity protection for the
>   client preamble, particularly the prefix header.*
> - *New protocols and schemes should not use this obfuscator.*

In practical terms:

- **RC4 is dead.** Every modern protocol design skips RC4 entirely.
  OSSH retains it because the obfuscator only protects the
  handshake; real session crypto is SSH's. Still, the obfuscator
  layer offers no integrity, so a censor with a known-plaintext
  attack on the preamble (e.g. the magic value's position) can
  inject without detection.
- **Fully-encrypted-traffic detection** flags bare OSSH. Without
  an OSSH prefix or a wrapping protocol, the wire is exactly the
  shape FET classifiers target.
- **Prefix layer is patch, not solution**. The prefix spec
  randomizes / mimics the *first packet*; steady-state is still
  high-entropy RC4-over-SSH and a sufficiently sophisticated
  censor can fingerprint that.
- **Magic value brute-forceability**. 4 bytes is small enough that
  a censor with enough computational budget can brute-force the
  obfuscator key for a given seed (RC4 keys are also small) and
  validate. Compute cost vs. value-of-detection makes this rarely
  worth it, but it's a theoretical concern.

For new design work, **don't model on OSSH**. Use it as the
"foundation that everyone built on but that everyone agrees is
end-of-life" reference point.

## Deployment Notes

- The substrate. Almost every Psiphon protocol-variant ID ends in
  `OSSH`. Disabling OSSH would disable most of Psiphon's stack;
  Psiphon's strategy is "wrap OSSH in a modern outer protocol" —
  TLS-OSSH, MEEK-OSSH, QUIC-OSSH, etc. — rather than replace OSSH.
- Bare `OSSH` (no prefix, no wrapping) is in `SupportedTunnelProtocols`
  per `protocol.go:225` but is rarely picked by the Psiphon
  tactics system in heavily-censored regions.
- Operator-side OSSH key rotation is bundled with Psiphon's
  server-entry redistribution; clients receive new keywords +
  server pubkeys via signed Psiphon network responses.
